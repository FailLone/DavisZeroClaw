## 1. Goal

Upgrade the article ingest pipeline so that:

1. **Extraction is model-grade, not heuristic-grade.** Replace the single `PruningContentFilter` with a multi-engine pipeline (trafilatura as default, OpenRouter-LLM as fallback, LLM-learned CSS selectors as the fast path per-host).
2. **Rust-side cleaning stops damaging content.** Fix `normalize_line` code-block flattening and the cross-document dedup that eats legitimate short repeats.
3. **Value scoring stops being hardcoded.** Replace the `0.55` baseline with multi-dimensional content signals (code density, link density, heading hierarchy, gopher-rule subset).
4. **The system learns per-site extraction rules by itself.** The existing TOML `[[sites]]` block is deleted; host-level rules are generated by an LLM from accumulated samples, with quality feedback driving re-learn.

Verbatim user direction this session:
- "我是想基于当前项目的 crawl4ai，增强其清洗的能力，让文章更可读，价值评分更准确"
- "我想放弃现有规则，直接跳到规则自学习"

---

## 2. Non-goals

- **No Reader-LM local deployment.** Requires GPU and carries CC-BY-NC-4.0 license constraints. OpenRouter-hosted LLM replaces it functionally.
- **No Jina Reader API.** It fetches the page itself, so it cannot use crawl4ai's authenticated profiles for 知乎/微信/淘宝.
- **No MCP server integration.** DavisZeroClaw is a daemon + worker pool; MCP is an agent-tool protocol; architectural mismatch.
- **No migration of existing TOML `[[sites]]` rules as seed data.** User explicitly chose "hard delete" (option A of three presented).
- **No schema change to `ArticleMemoryAddRequest`** or any existing public API surface. All changes are additive or internal.
- **No change to iMessage / Shortcut / CLI entry points.** The `POST /article-memory/ingest` endpoint and `articles ingest` CLI are unchanged on the caller side.
- **No user-provided fine-grained region feedback (Level 3).** Only implicit feedback and one-tap 👍/👎 are in scope.
- **No per-article rule overrides.** Rules are host-level only (domain suffix match, as today).

## 3. Constraints

- Must keep existing crawl4ai HTTP adapter architecture (`crawl4ai_adapter/server.py`) and Rust supervisor/profile-lock model (`src/crawl4ai_supervisor.rs`, `Crawl4aiProfileLocks`).
- Must keep persistence patterns: full-JSON write on every state change, `runtime/article_memory/` as the only runtime data root.
- Must keep deadlock invariants from the baseline spec: profile lock and queue lock never nested.
- Must keep zero-fork principle: `add_article_memory → normalize_article_memory → upsert_embedding` downstream stays byte-identical to CLI `articles add`.
- Local-only branch policy is in force: commit to the current branch, no push, no PR (per user directive on 2026-04-24).
- Rust files stay ≤ 400 lines; new modules split along cohesion lines.

## 4. Architecture Overview

### 4.1 Five Principles

1. **One rule surface.** Delete TOML `[[sites]]`. All host-level rules live in `runtime/article_memory/learned_rules.json`, written by the learning worker, with a single hand-written override file for emergencies.
2. **Tiered extraction.** Three engines (`learned-rules` → `trafilatura` → `openrouter-llm`) with cost increasing per tier. Pre-routing picks the starting tier; quality gate upgrades on failure.
3. **Structural preservation in Rust.** Cleaning functions respect fenced code blocks, list indentation, and near-window semantics instead of global normalization.
4. **Content signals over token counts.** Deterministic value scoring reads `code_density`, `link_density`, `heading_depth`, `list_ratio`, `avg_paragraph_chars` — not just `chars < min`.
5. **Self-healing rules.** Extraction quality is judged by the LLM on every ingest (same API call as value judge). Repeated partial/poor results mark a rule stale; stale rules trigger re-learning on next ingest.

### 4.2 Engine Tiers

| Tier | Engine | When used | Cost / article |
|------|--------|-----------|----------------|
| 0 | `learned-rules` | host has a non-stale entry in `learned_rules.json` | ~$0 (CSS selector extract) |
| 1 | `trafilatura` | cold-start default, fallback when Tier 0 stale | ~$0 (pip lib) |
| 2 | `openrouter-llm` | Tier 1 failed quality gate (upgrade ladder) | ~$0.002–0.03 |
| — | `pruning` (deleted) | n/a | crawl4ai's built-in, no longer used |

### 4.3 Phase Split

- **Phase 1** (4–5 days): engine pluggability + quality gate + `[[sites]]` deletion. Ships even without Phase 2.
- **Phase 2** (11–15 days): Rust cleaning fixes + multi-dim scoring + LLM `extraction_quality` field + rule self-learning loop + warmup CLI.

Phase 1 is independently shippable. Phase 2 must ship as a unit (the learning loop depends on the LLM `extraction_quality` field, which depends on multi-dim scoring, which depends on cleaning fixes).

## 6. Data Shapes

### 6.1 `LearnedRule`
```rust
struct LearnedRule {
    host: String,
    version: String,              // RFC3339 timestamp of learning event

    remove_selectors: Vec<String>,
    title_selector: Option<String>,
    start_markers: Vec<String>,
    end_markers: Vec<String>,
    confidence: f32,              // LLM-reported 0..1
    reasoning: String,             // for quarantine/debug

}
```

### 6.2 `RuleStats`
```rust
struct RuleStats {
    host: String,
    rule_version: String,
    hits: u64,
    partial: u64,
    poor: u64,
    consecutive_issues: u32,
    last_relearn_trigger: Option<String>,  // "consecutive_issues" | "poor_rate" | "manual"

}
```

### 6.3 `ContentSignals`
```rust
struct ContentSignals {
    code_density: f32,         // (chars in fenced/inline code) / total chars
    link_density: f32,         // link count / word count
    heading_depth: usize,       // max heading level present (1..6)
    heading_count: usize,
    list_ratio: f32,           // list lines / total lines
    avg_paragraph_chars: usize,
    alpha_ratio: f32,          // alphabetic chars / total
    symbol_ratio: f32,         // hash/punct chars / total
}
```

### 6.4 `GateResult`
```rust
struct GateResult {
    pass: bool,
    hard_fail_reasons: Vec<&'static str>,  // triggers rule_samples push
    soft_fail_reasons: Vec<&'static str>,  // triggers engine upgrade only

}
```

```json
{

  "translation_needed": bool,
  "extraction_quality": "clean" | "partial" | "poor",
```

## 7. Data Flow

### 7.1 Happy path (learned rule hit)
```
1. URL submitted → IngestQueue.submit() → 202
2. Worker picks job; pick_engine(host) returns EngineChoice::LearnedRules(rule)
3. crawl4ai_crawl(url, engine="learned-rules", learned_rule=rule) → markdown
4. QualityGate::assess → pass
5. Rust cleaning (structure-preserving) → normalized.md
6. deterministic_score (multi-dim) + judge_article_value (with extraction_quality)
7. extraction_quality="clean" → stats.hits += 1
8. decision="save" → upsert_embedding
9. finish(Saved) + iMessage
Total engine calls: 1 crawl + 1 LLM judge.
```

### 7.2 Learned rule stale (rediscovery path)
```
1. pick_engine returns EngineChoice::Trafilatura (rule was stale)
2. crawl4ai_crawl(engine="trafilatura") → markdown
3. QualityGate::assess → hard_fail (e.g., markdown_chars<500)
4. Persist HTML to rule_samples/<host>/
5. Upgrade: engine=openrouter-llm, crawl4ai_crawl again → markdown
6. QualityGate::assess → pass
7. Rust cleaning + judge → save
8. rule_samples/<host>/ now has N samples → flag ready
9. Hourly worker picks it up: learn new rule → validate → save to learned_rules.json
10. Next URL on same host: Tier 0 learned-rules hit
```

### 7.3 LLM reports poor extraction (even though gate passed)
```
1. pick_engine → LearnedRules (non-stale)
2. extract → pass gate
3. Rust cleaning
4. LLM judge returns extraction_quality="poor"
5. Force decision="candidate" regardless of value_score
6. mark_rule_stale(host); save_rule_sample(html); stats.poor += 1
7. Next URL on same host: pick_engine sees stale flag → Tier 1 trafilatura
8. Continues rediscovery flow (7.2 step 3+)
```

### 7.4 Rule learning worker (hourly)
```
for host in list_ready_hosts(rule_samples_dir, threshold=3):
    samples = load_latest_n(host, n=3)
    dom_outlines = [simplify_dom(s.html_snapshot) for s in samples]
    prompt = build_learn_prompt(host, dom_outlines, samples)
    rule_json = call_llm(learning_model, prompt)   # ~$0.02-0.05
    validation = validate_rule(rule_json, samples)
    if validation.ok:
        learned_rules[host] = LearnedRule { ...rule_json, stale: false }
        persist_learned_rules()
        clear_samples(host)
        stats[host] = RuleStats { hits: 0, partial: 0, poor: 0, ... }
    else:
        write(quarantine_rules/<host>-<ts>.json, rule_json + validation.errors)
        if config.notify_on_quarantine:
            imessage_send("规则学习失败: {host} — {primary_error}")
```

### 7.5 Error taxonomy

| Source | issue_type | stage | Recoverable? |
|--------|-----------|-------|--------------|
| crawl4ai http 503 | `crawl4ai_unavailable` | fetching | retry ladder |
| crawl4ai 504 | `crawl_timeout` | fetching | retry once |
| trafilatura returns empty | `extract_empty` | fetching | upgrade engine |
| openrouter-llm HTTP error | `extract_llm_failed` | fetching | fail job |
| QualityGate hard_fail | `quality_gate_hard_fail` | fetching | upgrade ladder, then fail if ladder exhausted |
| QualityGate soft_fail | `quality_gate_soft_fail` | fetching | upgrade only |
| Rule learning LLM error | `rule_learn_failed` | async | quarantine, keep old rule |
| Rule validation failed | `rule_validation_failed` | async | quarantine, keep old rule |

## 8. Quality Gate Specification

### 8.1 Hard-fail signals (trigger engine upgrade + rule sample capture)
- `markdown_chars < quality_gate.min_markdown_chars` (default 500)
- `kept_ratio < quality_gate.min_kept_ratio` (default 0.05)
- `paragraph_count < quality_gate.min_paragraphs` (default 3)
- `link_density > quality_gate.max_link_density` (default 0.5)

### 8.2 Soft-fail signals (trigger engine upgrade only, no sample capture)
- `code_block_ratio < 0.01` AND HTML `<code>/<pre>` tag ratio > 0.1 (code was flattened)
- `heading_count < 1` AND HTML had `<h1-6>` tags (structure lost)
- `boilerplate_markers` present in markdown (soft)

### 8.3 Upgrade ladder
```
learned-rules → trafilatura → openrouter-llm → Failed("quality_gate_rejected")
trafilatura   → openrouter-llm → Failed
openrouter-llm → Failed (no further tier)
```

Each upgrade increments `IngestJob.attempts`. Max 2 upgrades per job.

### 8.4 Cost circuit breaker
- Track daily count of `openrouter-llm` calls in `runtime/article_memory/llm_usage.json`.
- If `daily_llm_upgrades > quality_gate.cost.max_llm_upgrades_per_day`: new upgrade requests to Tier 2 are rejected with `Failed("llm_budget_exhausted")` until next UTC day.
- Soft-budget warnings (not enforced): log `tracing::warn!` when daily spend estimate exceeds `daily_llm_budget_usd`.

## 9. Rule Learning Specification

### 9.1 DOM simplification algorithm
Input: raw HTML (typically 10K–200K chars).
Output: tree outline, typically 500–2000 chars.

Per node: keep `tag`, `id` (if any), `class` (first two classes only), child element count. Skip all text, attributes, scripts, styles. Depth-limit at 8 levels; children-limit at 10 (show "... (N more)" elision).

Example output:
```
body
  header.site-header
  main.container
    article.post#post-123
      h1.post-title
      div.post-meta
      div.post-content
        p (×15)
        pre.code-block (×3)
        h2 (×4)
    aside.related
      ul.related-list
  footer.site-footer
```

### 9.2 Learning prompt structure
System: `"You are a web extraction rule generator. Given simplified DOM outlines from multiple pages of the same site, emit CSS selectors that would cleanly extract the main article content. Return strict JSON only."`

User:
```
Host: {host}
Failure context: {summary of why rule is needed}

Sample 1 DOM:
{simplified_tree_1}
Sample 1 first 500 chars of bad extraction:
{markdown_preview_1}

Sample 2 DOM: ...
Sample 3 DOM: ...

Emit JSON:
{
  "content_selectors": ["primary selector for article body", ...],
  "remove_selectors": ["selectors to drop noise blocks", ...],
  "title_selector": "selector for article title or null",
  "start_markers": ["optional text markers to start from", ...],
  "end_markers": ["optional text markers to end at", ...],
  "confidence": 0.0..1.0,
  "reasoning": "brief explanation of choices"
}

Prefer selectors that appear in all three samples. Use stable tag+class combinations, avoid :nth-child and dynamic IDs.
```

### 9.3 Validation
For each of the 3 samples:
1. Apply `rule.content_selectors[0]`; if empty, try next; fail if all empty.
2. Apply `rule.remove_selectors` to strip.
3. Apply `rule.start_markers` / `end_markers` on resulting text.
4. Run `QualityGate::assess` on the resulting markdown.
5. Collect: did gate pass? What's `markdown_chars` median?

Rule passes validation iff:
- All 3 samples yield non-empty extraction
- At least 2 of 3 pass the quality gate
- Median `markdown_chars` ≥ 1500

### 9.4 Stale triggers
A learned rule is marked `stale: true` when:
- `consecutive_issues >= rule_learning.stale_after_consecutive_issues` (default 2), where an "issue" is LLM judge `extraction_quality` in `{partial, poor}`, or
- `poor / (hits + partial + poor) > 0.3` after at least 10 hits, or
- Manually via `articles rule-learn mark-stale <host>`.

### 9.5 Warmup CLI
```
daviszeroclaw articles rule-learn warmup --from-existing [--hosts host1,host2] [--per-host 3]
```
- Reads `article_memory_index.json`, groups by host
- Selects N articles per host
- Re-fetches HTML via crawl4ai_adapter
- Writes directly to `rule_samples/<host>/` (bypassing the normal ingest path)
- Triggers immediate learning pass (not waiting for hourly scheduler)

### 9.6 Quarantine
- Path: `runtime/article_memory/quarantine_rules/<host>-<timestamp>.json`
- Content: rejected rule JSON + validation errors + sample references
- On write, if `rule_learning.notify_on_quarantine = true`: send iMessage to the primary `reply_handle` (if configured at daemon level).
- Existing (pre-quarantine) learned rule remains active; no downgrade to trafilatura.

## 11. Value Scoring Upgrade Specification

### 11.1 Multi-dimensional deterministic score
Replace `deterministic_value_report` (pipeline.rs:462) with:
```rust
fn deterministic_score(signals: &ContentSignals, matched_topics: &[String]) -> f32 {
    let mut s = 0.5;
    s += (matched_topics.len() as f32 * 0.05).min(0.15);
    if (0.05..0.30).contains(&signals.code_density) { s += 0.08; }
    if signals.heading_depth >= 2 { s += 0.05; }
    if signals.heading_depth >= 4 { s += 0.03; }
    if signals.link_density > 0.3 { s -= 0.10; }
    if (300..1500).contains(&signals.avg_paragraph_chars) { s += 0.05; }
    if signals.list_ratio > 0.5 { s -= 0.05; }
    s.clamp(0.0, 1.0)
}
```

### 11.2 Gopher-rule hard reject
```rust
fn gopher_reject(signals: &ContentSignals, markdown: &str) -> Option<&'static str> {
    if signals.link_density > 0.6 { return Some("link_density_too_high"); }
    if signals.alpha_ratio < 0.3 { return Some("alphabetic_ratio_too_low"); }
    if signals.symbol_ratio > 0.1 { return Some("symbol_ratio_too_high"); }
    None
}
```
If `gopher_reject` returns Some, skip LLM judge; decision is immediately `reject` with the returned reason.

### 11.3 LLM judge extension
Existing prompt in `pipeline.rs:419`; append:
```
You also act as an extraction-quality judge. Assess whether the article was cleanly extracted. Look for:
- Abrupt truncation or missing continuation
- Boilerplate UI text (nav, share, comments, "related articles") mixed into body
- Broken or flattened code blocks, lists, tables
- Wrong content region (e.g., only a comment instead of the article body)

Emit `extraction_quality`: "clean" (default; no visible issues), "partial" (minor noise but usable), or "poor" (would need re-extraction).
If partial or poor, emit `extraction_issues` (list of short codes) and `rule_refinement_hint` (brief free-text).
Default to "clean" unless there is a clear, specific issue.
```

Parsing: extend `parse_value_judge_response` to read the three new fields; all default to conservative values (`"clean"`, `[]`, `""`) if missing, so old responses still parse.

### 11.4 Feedback loop to LearnedRuleStore
After `judge_article_value` returns:
```rust
match report.extraction_quality.as_deref() {
    Some("poor") => {
        decision = "candidate".into();  // override
        learned_rules.mark_stale(&host, "llm_reports_poor");
        rule_samples.push(host, html, markdown, "llm_poor");
    }
    Some("partial") => {
        rule_stats.bump(&host, RuleEvent::Partial);
        if rule_stats.consecutive_issues(&host) >= config.stale_after_consecutive_issues {
            learned_rules.mark_stale(&host, "consecutive_partial");
        }
    }
    _ => {
        rule_stats.bump(&host, RuleEvent::Hit);
    }
}
```

## 12. Concurrency & Persistence

### 12.1 New lock ordering
Two new locks (both `tokio::sync::Mutex`):
- `LearnedRuleStore`: read-heavy. Use `RwLock` instead.
- `RuleStats`: write-heavy. Mutex is fine; short critical sections.
- `SampleStore`: filesystem-backed, per-host subdirectory. No in-memory lock; use `flock` advisory lock on `rule_samples/<host>/.lock` if parallel writes possible.

**Invariant**: `QueueState lock > profile lock > learned_rules RwLock > rule_stats Mutex`. Never acquire in reverse. Rule learning worker acquires `learned_rules` last (write mode), does not hold queue lock.

### 12.2 Persistence paths (additive)
```
runtime/article_memory/
├── (existing files unchanged)
├── learned_rules.json            ← RwLock-protected, atomic write via tempfile
├── learned_rules_stats.json      ← Mutex-protected, atomic write
├── rule_samples/<host>/<ts>-<job_id>.json  ← append-only files
├── rule_samples/<host>/<ts>-<job_id>.html  ← HTML snapshot
├── quarantine_rules/<host>-<ts>.json
└── llm_usage.json                ← daily LLM call counter, resets at UTC midnight
```

Atomic write: use `tempfile::NamedTempFile::new_in(parent).persist(target)` for all files above. This also fixes the pre-existing non-atomic `fs::write` in `ingest/queue.rs` (called out in a follow-up of the baseline spec).

### 12.3 Rule learning worker scheduling
Spawned at daemon start alongside `IngestWorkerPool`:
```rust
let learning_worker = RuleLearningWorker::spawn(
    paths.clone(), learned_rules.clone(), rule_stats.clone(),
    sample_store.clone(), supervisor.clone(), config.rule_learning.clone(),
);
```
Internal loop: `tokio::time::interval(Duration::from_secs(3600))`. Each tick:
1. `sample_store.list_ready_hosts(threshold=3)`
2. For each host (sequentially, not parallel — LLM cost): attempt learning.
3. Sleep until next tick.

Startup behavior: on first tick after daemon start, immediately scan (do not wait 1 hour).

## 19. Glossary

- **Engine**: a content extraction strategy (`learned-rules`, `trafilatura`, `openrouter-llm`, deprecated `pruning`).
- **Quality gate**: deterministic post-extract check that decides pass / hard-fail / soft-fail.
- **Hard fail**: statistical signals so bad that the HTML is worth keeping as a learning sample.
- **Soft fail**: structural signals suggesting poor extract, but not worth a learning round.
- **Upgrade ladder**: the engine progression walked when a quality gate fails.
- **Learned rule**: a host-level JSON describing CSS selectors to extract article body, generated by the learning worker.
- **Rule sample**: a single HTML snapshot + failure context, waiting for its host to accumulate enough siblings to trigger learning.
- **Stale**: a learned rule flagged for re-generation on next ingest for its host.
- **Quarantine**: a learned rule that failed validation and is held aside for manual review.
- **DOM outline**: simplified tag+class tree used as LLM input during rule learning.
- **Extraction quality**: LLM-reported 3-level verdict (`clean` / `partial` / `poor`) on whether an extraction preserves intent.
- **Gopher rules**: text-density heuristics from the Gopher LLM dataset paper, used as cheap hard-reject signals.
