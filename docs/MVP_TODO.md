# DavisZeroClaw MVP Todo List

最后更新：2026-03-22
负责人角色：架构师（不直接开发，只负责拆解、审阅、派发 sub-agent）

## 1. 执行原则

1. MVP 必须交付一个可现场演示的 demo，不能只有代码、测试或架构图。
2. 每落地一个较大的功能，必须安排一名 sub-agent 扮演“主任工程师”进行审核和挑刺，重点检查：
   - 潜在 bug
   - 边缘 case
   - 丑陋代码和可维护性问题
   - 验收口径是否真实可测
3. 每次任务结束后，本文件必须更新状态、风险和下一步。
4. 本阶段优先做“端到端跑通”，不是做“大而全”。

## 2. sub-agent 组织方式

- 架构师：负责范围控制、任务拆解、验收口径、审查安排，不直接承担业务代码开发。
- 实现类 sub-agent：按任务分工负责具体落地，每次只拥有一个清晰的交付目标。
- 主任工程师 sub-agent：在每个大功能完成后独立审查，优先找 bug、边缘 case、脆弱设计和难维护实现。

标准推进顺序：

1. 架构师从本文件中选择一个 `进行中` 或 `未开始` 的高优任务。
2. 指派 1 名实现类 sub-agent 完成该任务。
3. 任务产出后，立即指派 1 名主任工程师 sub-agent 进行审查。
4. 根据审查结论决定：通过、返工、或降级范围。
5. 回写本文件中的状态、风险、下一步。

## 3. MVP 范围冻结

### 3.1 本期必须做

- macOS 宿主可启动，并具备基础配置能力。
- 1 条可演示的 Siri/Shortcuts -> Gateway -> Agent -> Home Assistant 控制链路。
- 1 个可演示的自然语言问答/查询能力。
- 1 条异步结果反馈链路（至少支持 Bark 推送或等价反馈方式）。
- 可用于演示的日志、配置样例、操作脚本和验收清单。

### 3.2 本期明确不做

- 云中继 / 外网访问。
- 40+ 本地生活服务全集成。
- 自我进化、习惯学习、自动生成 HA 自动化规则。
- 桌面视觉自动化 / Computer Use。
- Linux 兼容性打磨。
- 多模型、多入口、多端并行适配。

## 4. Demo 成功标准

以下场景至少要稳定演示通过 3 次：

1. 用户通过 Siri 或 Shortcuts 发送“打开客厅灯”，宿主成功调用 Home Assistant 执行，并把结果反馈到手机。
2. 用户发送一个自然语言查询请求，系统返回明确结果，而不是空泛闲聊。
3. 在 Home Assistant 不可达、Token 错误或模型超时时，系统能给出明确失败反馈，不静默挂死。

## 6. 大功能拆分与审查关卡

以下事项视为“大功能”，每项完成后都必须安排一次“主任工程师审查”：

1. Gateway 接入层
2. LLM Provider 接入
3. Home Assistant 控制闭环
4. 手机反馈闭环
5. 查询型工具能力
6. Demo 联调与交付包装

每次审查至少要回答 4 个问题：

1. 有没有会导致错误控制设备的风险？
2. 有没有超时、重试、重复触发、并发触发问题？
3. 有没有“看起来能跑，但现场 demo 会翻车”的脆弱点？
4. 有没有为了赶工堆出来的丑陋实现需要在 MVP 内收敛？

## 8. 当前已识别风险

- 当前文档把“本地生活服务 MCP 集群”和“自我进化”写得很大，但它们不适合进入首个可演示 MVP。
- 当前文档没有把 demo 场景、失败场景和验收方式写成可执行清单，容易出现“代码完成但无法稳定演示”。
- 当前文档没有明确 Home Assistant 实体白名单策略，存在误控风险。
- 当前文档没有明确请求去重、超时和回执机制，现场联调时很容易出现 Siri 已结束但后台无反馈的问题。
- 当前文档没有明确配置装载与密钥管理方式，实际落地时第一步就会卡住。
